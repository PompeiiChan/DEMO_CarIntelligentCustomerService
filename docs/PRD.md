# 产品需求文档（PRD）

**项目名称**：汽车智能客服系统
**客户**：上汽大众售后客服体系
**版本**：V1 MVP
**最后更新**：2025-06-09（整合参考工单系统机制）

---

## 1. 背景与调研结论

### 1.1 项目背景

**客户背景**：上汽大众售后客服体系

**核心痛点**：
1. 人工客服压力大：大量重复性、标准化咨询占用人工时间
2. 传统客服机器人体验差：关键词匹配+规则树，理解能力差
3. 服务成本高：每一通人工电话成本约10-20元，7×24小时覆盖需要大量人力

**解决目标**：
- 用AI接管60-70%的高频、标准化咨询
- 通过多轮对话+上下文管理收集完整信息
- 精确转人工：只在用户明确要求或复杂场景时转人工

**落地渠道**：Web Demo

---

### 1.2 目标用户

| 用户类型 | 场景 | 需求 |
|---------|------|------|
| 潜在购车客户 | 售前咨询 | 车型对比、价格、优惠、试驾预约 |
| 已订车客户 | 售中履约 | 订单状态、生产进度、物流跟踪、交付时间 |
| 已购车车主 | 售后服务 | 用车指导、保养提醒、故障排查 |
| 客服人员 | 人工服务 | 接手AI转来的对话、查看上下文、标注问题案例 |
| 管理员 | 系统管理 | 知识库管理、意图配置、监控统计 |

---

### 1.3 使用场景

**用户使用场景**：
- 7×24小时，随时随地
- Web端Demo展示完整链路

**核心使用流程**：
1. 用户打开对话页 → 快捷操作/输入问题
2. AI识别意图 + 情绪 → 检索知识库 → 生成回复
3. 若无法解决 → 转人工 → 坐席接手
4. 坐席查看AI历史对话 → 回复用户 → 转回AI

---

### 1.4 竞品调研结论

**竞品分析**：
- **比亚迪**：24/7 AI客服，100%实时响应
- **网易七鱼**：80%问题解决率，"一触即达"点击交互
- **智齿科技**：87%+ AI准确率，全渠道覆盖
- **阿里云Beebot**：通义千问大模型，快速学习文档

**可借鉴点**：
- ✅ 7×24小时AI客服 + 人工转接
- ✅ 意图识别 + 置信度判断
- ✅ 点击式快捷交互
- ✅ 多轮对话上下文管理
- ✅ 87%+ AI准确率作为技术目标

**不采用点**：
- ❌ 全渠道覆盖（Demo专注Web）
- ❌ 复杂技能组管理
- ❌ 300+数据报告（Demo精简指标）

#### Agent节点列表示例

选中某个Response后，右侧显示节点列表（按顺序）：

```
[✓] 情绪识别 (50ms)

  - 输出: {emotion: "neutral", confidence: 0.3}
  - 状态: 已完成

[✓] 意图识别 (120ms)
  - 输入: "我的车发动机异响"
  - 输出: {intent: "维修咨询", confidence: 0.85}
  - 状态: 已完成

[✓] 置信度判断 (30ms)
  - 输入: {confidence: 0.85}
  - 输出: "高置信度，进入知识检索"
  - 状态: 已完成
```

## 5. 关键业务逻辑

### 5.1 对话状态流转

```
空闲等待 → AI对话中 →
[连续3次未匹配/负面情绪/用户要求/复杂场景] →
创建工单 → 工单排队等待分配 →
坐席接单（工单状态：处理中） → 人工服务中 →
[对话结束/转回AI] →
关闭工单（工单状态：已完成） → 空闲等待
```

**工单状态流转**：
```
待分配 → 已分配 → 处理中 → 已解决 → 已完成
```

**工单优先级规则**：
- 高优先级：负面情绪、事故场景、投诉关键词
- 中优先级：连续3次未匹配、复杂维修
- 低优先级：用户主动要求转人工

### 5.2 意图识别规则

- **高置信度**（>70%）：直接回复
- **中置信度**（40-70%）：追问澄清
- **低置信度**（<40%）：引导用户重新描述

### 5.3 转人工触发条件

1. **用户主动要求**：说"转人工""人工客服"
2. **AI连续3次未匹配**：连续3次置信度<40%
3. **用户情绪负面**：负面情绪概率>60%
4. **复杂场景**：涉及重大维修、事故、投诉（关键词匹配）

### 5.4 知识检索优先级（混合检索策略）

**V1 MVP**：结构化数据（Mock） > Markdown向量检索 > 历史QA
- 结构化数据查询：订单状态、车型参数、价格信息（SQL查询）
- Markdown向量检索：保养知识、维修指引、政策文档（Chroma向量检索）
- 历史QA匹配：高频问答对（关键词匹配）

**V2+**：结构化数据 > Markdown文档 > 历史QA > 实时计算（价格估算、贷款计算）

## 6. 关键功能实现思路

### 6.1 对话交互Pipeline

**触发场景**：用户在C01客户端对话页输入问题后

**实现思路**：
```
用户输入 → 情绪识别 → 意图识别（CICC框架） → 置信度判断 →
[高置信度→知识检索→回复生成] / [中置信度→追问澄清] / [低置信度→引导重述] →
转人工判断 → 输出回复 → 上下文记忆更新
```

**关键技术选型**：
- **情绪识别**：通义千问API（8B-32B）
- **意图识别**：通义千问API + 用户提供的意图分类体系
- **知识检索**：Markdown文档 + Chroma向量检索
- **上下文管理**：最近5轮对话存储在内存/Redis

---

### 6.2 意图识别（CICC框架）

**实现思路**：
- **输入**：用户当前问题 + 最近5轮对话上下文
- **分类**：6种核心意图（车型咨询/价格咨询/试驾预约/维修咨询/故障排查/保养咨询）
- **置信度判断**：
  - 高置信度（>70%）：直接进入知识检索
  - 中置信度（40-70%）：追问澄清
  - 低置信度（<40%）：引导用户重新描述
- **纠错机制**：用户说错时主动澄清

**关键技术选型**：
- **模型**：通义千问（8B-32B，硅基流动API）
- **提示词**：Agent生成默认提示词模板，用户可后续调整
- **示例句**：用户提供每个意图的10-20个示例句

---

### 6.3 知识检索（混合检索策略）

**实现思路**：
```
意图提取 →
[结构化数据查询 OR Markdown向量检索 OR 历史QA匹配] →
结果排序 → 答案生成 → 引用标注
```

**检索路径选择**：
1. **结构化数据查询**（订单、车型、价格）：
   - 识别意图为"订单查询/车型咨询/价格咨询"
   - 提取查询条件（订单号、车型名称、价格区间）
   - 执行SQL查询：`SELECT * FROM orders WHERE order_id = ?`
   - 返回结构化数据

2. **Markdown向量检索**（保养、维修、政策）：
   - 识别意图为"维修咨询/故障排查/保养咨询"
   - 提取关键词，进行向量检索
   - Chroma检索Top-5文档片段
   - 返回文档片段

3. **历史QA匹配**（高频问答）：
   - 关键词匹配历史QA库
   - 返回匹配的问答对

**关键技术选型**：
- **向量检索**：
  - Embedding模型：通义千问Embedding
  - 向量库：Chroma（持久化到本地文件）
- **结构化数据查询**：
  - 数据库：SQLite
  - 查询方式：SQL查询语句
- **文档处理**：Markdown文档分块（chunk_size=500，overlap=50）
- **排序规则**：相似度评分 + 意图匹配度 + 数据源优先级

---

### 6.4 情绪识别

**实现思路**：
- **输入**：用户当前问题
- **分类**：正面/负面/中性
- **阈值判断**：
  - 负面情绪概率>60%：触发安抚/转人工
  - 负面情绪概率40-60%：记录但不触发
  - 负面情绪概率<40%：正常处理

**关键技术选型**：
- **模型**：通义千问API（与意图识别共用）
- **提示词**：Agent生成默认提示词，判断情绪分类

---

### 6.5 转人工链路（工单机制）

**实现思路**：
```
触发条件判断 →
[连续3次未匹配 OR 情绪负面 OR 用户要求 OR 复杂场景] →
创建工单（工单号、优先级、转接原因） →
工单排队等待分配 →
传递上下文（用户问题、AI尝试、转接原因、用户信息） →
S01坐席工作台接收工单 →
坐席接单（工单状态：处理中） →
AI静默 →
坐席处理用户问题 →
[转回AI OR 关闭工单] →
工单状态更新（已解决/已完成）
```

**工单数据结构**：
```json
{
  "ticket_id": "T20250609001",
  "user_id": "U123456",
  "priority": "high|medium|low",
  "status": "pending|assigned|processing|resolved|completed",
  "reason": "连续3次未匹配|情绪负面|用户要求|复杂场景",
  "context": {
    "user_questions": ["问题1", "问题2", "问题3"],
    "ai_attempts": ["尝试1", "尝试2", "尝试3"],
    "intent": "维修咨询",
    "emotion": "negative"
  },
  "created_at": "2025-06-09T10:00:00Z",
  "assigned_at": "2025-06-09T10:01:00Z",
  "resolved_at": null,
  "completed_at": null,
  "agent_id": "A001",
  "processing_duration": 300
}
```

**上下文传递**：
- 用户基本信息（用户ID、昵称、历史工单数）
- 最近5轮对话历史
- AI尝试解决的问题
- 转人工原因（哪个触发条件）
- 当前意图分类
- 用户情绪状态

**关键技术选型**：
- **消息传递**：WebSocket实时推送
- **状态存储**：SQLite存储工单数据，Redis缓存对话状态
- **队列管理**：FIFO队列，按优先级排序，坐席按顺序接手
- **工单分配**：自动分配给空闲坐席，支持负载均衡

### 6.6 上下文管理与用户画像

**实现思路**：
- **记忆窗口**：最近5轮对话
- **存储结构**：role（user/assistant）、content、intent、timestamp
- **窗口滚动**：超过5轮后，最老的对话被移除
- **过期时间**：2小时无活动后清空上下文
- **用户画像持久化**：存储用户基本信息、历史会话统计、偏好标签

**用户画像数据结构**：
```json

  "ticket_count": 2,
  "intents": ["车型咨询", "价格咨询"],
  "preference_tags": ["关注价格", "喜欢SUV"],
  "satisfaction_score": null
}
```

**关键技术选型**：
- **存储**：SQLite存储用户画像，Redis缓存会话上下文
- **传递**：转人工时，上下文作为JSON传给坐席端
- **更新**：每次对话后更新用户画像（会话次数、意图分布）

### 6.7 回复生成

**实现思路**：
- **输入**：检索到的Top-5文档片段 + 用户问题 + 对话上下文
- **生成**：
  1. 基于知识片段生成答案
  2. 标注引用来源
  3. 追问补充信息（如果信息不全）
  4. 引导下一步操作

**关键技术选型**：
- **模型**：通义千问API（与意图识别共用）
- **提示词**：Agent生成默认提示词模板

### 6.8 售后维修场景

**实现思路**：

```
症状分类（警告灯/异响/功能失效/事故） →
[事故/无法启动/安全相关 → 直接触发道路救援] →
[其他 → 知识检索 → 解决方案推荐] →
若无法解决 → 转人工
```

**症状分类**：
- 警告灯：发动机故障灯、机油灯、刹车系统灯等
- 异响：发动机异响、底盘异响、车身异响
- 功能失效：空调失效、车窗失效、中控失效
- 事故：碰撞、无法启动、安全相关

**关键技术选型**：
- **关键词匹配**：事故关键词库（"事故、碰撞、撞了、无法启动"）
- **优先级判断**：安全相关 > 道路救援 > 常规维修

2. Markdown向量检索（Chroma）：保养知识、维修指引、政策文档
3. 历史QA匹配（关键词）：高频问答对

**检索路径选择规则**：
- 意图为"订单查询/车型咨询/价格咨询" → 结构化数据查询
- 意图为"维修咨询/故障排查/保养咨询" → Markdown向量检索
- 其他意图 → 历史QA匹配
